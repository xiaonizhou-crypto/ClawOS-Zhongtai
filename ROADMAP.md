# Roadmap

ClawOS-Zhongtai 现在还是一个非常早期的产品原型。

但方向已经很清楚：
它不是往“更多 agent 页面”长，
而是往“更完整的组织中台”长。

---

## Phase 1 — 清楚地长成一个产品入口

目标：让第一次点进仓库的人，能快速理解这不是普通 agent dashboard。

当前重点：
- 双语 README
- 核心截图
- 产品定位
- Features / Use cases / FAQ
- GitHub 基础门面

结果标准：
- 外部用户第一次看到仓库，就能说清楚它是干什么的
- 不再像实验文档，而像一个正在成形的产品

---

## Phase 2 — 建立更完整的产品叙事

目标：从“能看懂”走到“能讲出去”。

计划补齐：
- `VISION.md`
- `ROADMAP.md`
- 品牌封面图
- 更完整的产品模块定义
- 更清晰的中英文口径统一

结果标准：
- 能对外讲清楚为什么这个方向成立
- 能让别人感受到这不是换壳，而是一个明确的新产品方向

---

## Phase 3 — 组织结构成形

目标：把“中台”从概念变成结构。

重点方向：
- Dashboard
- Roster
- Marketplace
- 组织角色体系
- 任务承接 / 分派 / 调度视图
- 可被编入组织的能力单元定义

结果标准：
- 系统不只是展示 activity，而是开始呈现角色、编制、职责和可调度关系

---

## Phase 4 — 三省六部表达层成熟

目标：让三省六部成为高质量的组织语言，而不是表面皮肤。

重点方向：
- 三省六部角色映射
- 组织语言统一
- 可解释的权责流转
- 不同页面的制度感一致
- 从 UI 命名到交互逻辑都更像一个组织系统

结果标准：
- 用户一看就能感到秩序感
- 不是“古风换皮”，而是“结构被一种组织语言准确表达”

---

## Phase 5 — 能力市场与组织编制联动

目标：让 agent / capability 不只是展示，而是真的能被组织吸纳。

重点方向：
- agent 能力卡
- 雇用 / 接入 / 编组逻辑
- 岗位视角 vs 能力视角切换
- marketplace 到 roster 的流转关系
- “可上架能力”与“在岗组织单元”的边界定义

结果标准：
- Marketplace 不只是陈列，而是成为组织扩张入口

---

## Phase 6 — 产品化与发布能力

目标：从原型走向可持续演进的产品仓库。

计划补齐：
- CHANGELOG
- Release notes
- CONTRIBUTING
- Issue templates
- 品牌资产管理
- 更明确的版本节奏

结果标准：
- 仓库开始具备持续发布和外部协作能力

---

## 长期方向

长期看，ClawOS-Zhongtai 不应该只是一种产品形态。

它更可能演化为：
- 一个多 Agent 组织层产品
- 一个承载不同角色体系的中台框架
- 一个把组织设计、能力编制、协作界面结合起来的系统

三省六部会是它的重要表达方向之一，
但不是唯一方向。

真正长期成立的，是下面这句话：

**让 Agent 系统开始拥有组织结构。**
